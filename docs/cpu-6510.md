# The 6510 / 6502 CPU

The C64's CPU is the **MOS 6510**, a 6502 core with an added 6-bit on-chip I/O
port (used to bank the ROMs and drive the cassette). For programming purposes
it *is* a 6502 — every 6502 reference applies, plus the banking port at
`$00/$01` and the C64's undocumented opcodes. It runs at ~**0.985 MHz (PAL)** /
~**1.023 MHz (NTSC)**, sharing the bus with the VIC-II (which steals cycles — see
[vic-ii.md](vic-ii.md)).

## Starter notes

### Programming model

- **Registers:** `A` (accumulator), `X`, `Y` (index), `SP` (stack pointer, page
  `$0100–$01FF`), `PC`, and the status register `P` (flags: N V - B D I Z C).
- **No multiply/divide, no 16-bit math.** Everything is 8-bit; you build wider
  arithmetic from `ADC`/`SBC` with the carry. `D` (decimal) flag enables BCD —
  **clear it (`CLD`) at startup**; the KERNAL doesn't guarantee it.
- **8-bit index registers** mean tables are naturally ≤256 bytes; larger data
  uses indirect-indexed addressing or self-modifying code.

### Addressing modes (the heart of 6502 optimization)

Immediate, zero page, zero page,X/,Y, absolute, absolute,X/,Y, indirect (`JMP`
only), **indexed-indirect `(zp,X)`**, and **indirect-indexed `(zp),Y`**. The last
one — `LDA ($fb),Y` — is the workhorse for walking through memory via a 16-bit
pointer stored in zero page.

### Zero page is precious

Page `$00` has **shorter, faster** instructions (2 bytes/3 cycles vs 3 bytes/4)
and is the *only* place you can store the 16-bit pointers used by `(zp),Y`.
BASIC and the KERNAL use most of it, but a handful of locations are free for
your own code (notably `$02`, `$fb–$fe`, `$22–$2a` region when BASIC is off). If
you bank out BASIC/KERNAL, much more opens up.

### Cycle counting

Instructions take 2–7 cycles. Two rules matter constantly:

- **Page-boundary penalty:** indexed reads (`abs,X` / `abs,Y` / `(zp),Y`) cost
  **+1 cycle** when the index crosses a 256-byte page. Branches cost +1 when
  taken, +1 more if the target is on another page.
- **A PAL frame is 312 rasterlines × 63 cycles ≈ 19,656 cycles**; NTSC is 263 ×
  65. That's your per-frame budget. Demos squeeze effects into it by
  *cycle-exact* code, often **"speedcode"** (fully unrolled, no loops).

| Range | Contents |
|-------|----------|
| `$0000/$0001` | 6510 I/O port (data dir / data) — controls ROM/RAM banking |
| `$0002–$00FF` | Zero page (BASIC `$02–$8F`, KERNAL `$90–$FF`) |
| `$0100–$01FF` | Hardware stack |
| `$0200–$03FF` | KERNAL/BASIC work area, input buffer, vectors |
| `$0400–$07FF` | Default **screen RAM** (1000 bytes) + sprite pointers at `$07F8` |
| `$0800–$9FFF` | BASIC program + variable RAM (free RAM if BASIC is off) |
| `$A000–$BFFF` | BASIC ROM (or RAM if banked out) |
| `$C000–$CFFF` | Free 4K RAM (never banked — handy for ML routines) |
| `$D000–$DFFF` | **I/O**: VIC-II `$D000`, SID `$D400`, color RAM `$D800`, CIA1 `$DC00`, CIA2 `$DD00` (or char ROM / RAM by banking) |
| `$E000–$FFFF` | KERNAL ROM (or RAM) |

Banking is controlled by `$01` (bits 0–2: LORAM/HIRAM/CHAREN). A common demo
setup is `LDA #$35 : STA $01` — RAM under BASIC and KERNAL, I/O still visible —
giving you `$A000–$FFFF` as RAM while keeping the chips reachable.

### KERNAL & BASIC ROM entry points

The KERNAL is a stable **jump table at `$FF81–$FFF3`**. The ones you'll use most:

| Vector | Name | Purpose |
|--------|------|---------|
| `$FFD2` | `CHROUT` | print a character (A) |
| `$FFCF` | `CHRIN` | read a character |
| `$FFE4` | `GETIN` | get a keypress (non-blocking) |
| `$FFBA`/`$FFBD`/`$FFD5` | `SETLFS`/`SETNAM`/`LOAD` | load files |
| `$FFD8` | `SAVE` | save |
| `$FFE1` | `STOP` | test RUN/STOP |
| `$FF81` | `CINT` | init screen editor |

The CPU also has fixed hardware vectors at the top of memory: **`$FFFA/B` NMI**,
**`$FFFC/D` RESET**, **`$FFFE/F` IRQ/BRK**. With KERNAL banked in, the RAM
vectors `$0314/5` (IRQ), `$0318/9` (NMI) are what you usually hook.

### Interrupts

- **IRQ** — maskable (`SEI`/`CLI`). By default fires ~60×/sec from CIA1 timer A
  (jiffy clock + keyboard scan). For effects you redirect it to a **raster
  interrupt** (the VIC-II asserts IRQ at a chosen scanline — see
  [vic-ii.md](vic-ii.md)). Acknowledge the source or it re-fires forever:
  for VIC raster IRQs write back to `$D019`; for CIA read `$DC0D`.
- **NMI** — non-maskable; from CIA2 (RESTORE key, RS-232). Demos often take it
  over to stop RUN/STOP-RESTORE from breaking the show.
- **Stable raster** — the timing jitter (0–7 cycles, because an interrupt waits
  for the current instruction to finish) is the central problem in raster
  effects; the **double-IRQ / NOP-slide** trick removes it. See the FLI write-up
  linked in [demoscene-effects.md](demoscene-effects.md).

### Illegal / undocumented opcodes

The NMOS 6510 has ~25 stable "illegal" opcodes that fall out of the silicon —
e.g. `LAX` (load A and X together), `SAX`, `DCP`, `ISC`, `SLO`, `RLA`, `ANC`,
`ALR`, `ARR`, and the unstable `SHX/SHY/SHA/TAS`. They're widely used in
size-coded intros and speedcode because they combine two operations in one
instruction. They work on real NMOS C64s and in accurate emulators; avoid the
unstable ones unless you know the hardware. Codebase64 and Groepaz's *No More
Secrets* document them exhaustively.

## Annotated resources

### Primary references

- **[C64 Programmer's Reference Guide](https://archive.org/details/c64-programmer-ref)**
  *(primary — official)*. The 1982/83 Commodore manual. Authoritative on the
  memory map, KERNAL routines, and BASIC. PDF/EPUB/searchable text on archive.org.
  **Local copy:** [`reference/c64-programmers-reference-guide.pdf`](reference/c64-programmers-reference-guide.pdf)
  (518 pp, searchable) + [`.txt`](reference/c64-programmers-reference-guide.txt) for grepping.
- **[Mapping the Commodore 64 (Sheldon Leemon)](https://www.zimmers.net/anonftp/pub/cbm/c64/manuals/mapping-c64.txt)**
  *(primary)*. The definitive annotated memory map — every important location
  `$0000–$FFFF` with what it does and how to use it. Zero page is split into
  BASIC working storage (`$0–$8F`) and KERNAL work area (`$90–$FF`).
- **[zimmers.net C64 memory map (C64.MemoryMap.txt)](https://www.zimmers.net/anonftp/pub/cbm/maps/C64.MemoryMap.txt)**
  *(primary)*. Compact canonical I/O register listing — VIC `$D000–$D02E`,
  SID `$D400–$D41C`, CIA1 `$DC00`, CIA2 `$DD00`.

### Instruction set & 6502 core

- **[masswerk 6502 instruction set](https://www.masswerk.at/6502/6502_instruction_set.html)**
  *(secondary, excellent)*. Searchable opcode table with cycle counts, flags,
  and addressing modes. The reference to keep open while coding.
- **[6502.org](http://6502.org/)** *(primary hub)*. Tutorials, the famous
  "Programming the 65816 / 6502" texts, source archive, and forums. The
  datasheet archive at `archive.6502.org/datasheets/` hosts the chip PDFs.
- **[Codebase64 — 6502/6510 coding index](https://codebase64.c64.org/doku.php?id=base:6502_6510_coding)**
  *(primary community hub)*. Tutorials, addressing-mode references, **illegal
  opcode** articles (ANC, ALR, SHX/SHY), optimization (speedcode, unrolled
  loops), and reusable routines.
- **[NMOS 6510 Unintended Opcodes (groepaz / hitmen)](http://www.oxyron.de/html/opcodes02.html)**
  *(reference)*. The oxyron opcode matrix plus links to *No More Secrets* — the
  exhaustive illegal-opcode + cycle-timing reference demo coders rely on.

### Quick lookups

- **[C64-Wiki](https://www.c64-wiki.com/)** — fast pages for any register,
  KERNAL routine, or concept (e.g. search "raster interrupt", "KERNAL").
